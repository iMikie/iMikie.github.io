Reflections on Version Control

Sharing the good stuff in the modern world. 

Version control was invented to solve a couple of problems.

 

First, it enables multiple people to work on the same set of files without stomping on each other's work.  For example, two or more may want to work on the same file at the same time.  If one copies his version of the file back to the master directory then the other person does, the latter over-rides the former.  You usually want only one person to be modifying a file at the same time. Version control catches modifications made by more than one person to the same file and assists in sorting out the differences

 

Second, you may need several versions of the set of files.  One is the last stable version, another is the current set of feature enhancements you are working on, another may be a special version for an important client.  If you simply copied the set of files each time you needed a new version then enhancements made to one version would have to be manually edited back into the other versions. 

 

Here is another example. I have written a science fiction story where the protagonist is a very bright 12 year old girl. (I have a very bright 12-year-old niece and can write like she talks). I finish the story and my friends read it and tell me they don’t think it’s realistic for a 12 year old.  So the first thing I did was create a version of the first chapter of the story where she’s 16. Most of the language that pegs her as 12 is in that chapter. Now I have two versions but they only differ in one chapter. 

 

Well pretty soon I have a folder full of versions and I have no idea which contains which changes.  I'm now using git and GitHub for that story.  GitHub is just a cloud site where you can store git repositories and pull and push changes. 

 

Using git I can break the file up into separate files for the chapters.  I make a 16 year-old-branch and a 12-year-old branch.  I can also edit the master or make another branch if I'm making changes that I want to appear in all versions -- I don't have to copy and paste between versions.  I can pull the changes down into the branches, but only when they are done.  That way if I have an opportunity to submit the story to a magazine, I always have a version that's ready to go.

 

Version control enforces a procedure so disasters don't happen.  Typically a good version control system can reproduce any of the previous versions

 

I used used version control before, one of the centralized kinds, maybe CVS or Subversion.  It was horrible, cryptic, and every so would get its drawers in a snit such that my head of engineering would have to spend a day getting everything sorted out and building again.  

 

Git is much better.  There's about 3 or 4 commands that you need to know and it just works.  There are a lot of resources on line so if you run into a problem, someone has solved it.

 

There is even somthing like GetFlow which is a scheme for how a large programming project should make branches, what they should be named and when and how they get merged back in.  This appears to have caught on in the community.  I appreciate the notion of being able to walk onto any project and already know how it's source control is organized.  SourceTree is a GUI for supporting Git Flow and the original GitFlow article by Vincent Driessen
